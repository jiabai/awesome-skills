# ARCHITECTURE.md 通用模板

适用于 Codex 和 Claude Code 的项目架构文档生成模板。

---

## 文件定位

这是一个**元规范**文件，定义"如何为任意项目编写 ARCHITECTURE.md"。

AI Agent 基于此模板为目标项目生成具体的架构文档，放置于项目根目录 `ARCHITECTURE.md`。

---

## 核心原则（不可协商）

违反以下原则会导致文档失效：

| 原则 | 说明 | 反面案例 |
|------|------|----------|
| **保持简短** | 每个贡献者都要读，越短越不易过时 | 写成 50 页详细文档 |
| **只写稳定内容** | 不写频繁变化的内容，一年更新几次即可 | 详细描述每个函数实现 |
| **不同步代码** | 不试图与代码保持同步，只在架构变化时修订 | 每次提交都更新文档 |
| **不链接** | 链接会失效，鼓励读者用符号搜索找实体 | 链接到具体文件路径 |
| **回答位置问题** | 必须回答"X 在哪？"和"这段代码做什么？" | 只写抽象概念描述 |

---

## 格式约束

| 约束项 | 规则 |
|--------|------|
| **文件位置** | 项目根目录，与 README.md、CONTRIBUTING.md 并列 |
| **标题层级** | 使用 `#` 和 `##`，标题后空一行 |
| **正文风格** | 简洁句子，避免长篇解释 |
| **命名实体** | 用完整名称（文件、模块、类型），不链接 |
| **代码地图** | 用简短描述而非详细实现 |

## 架构文档骨架模板

```
# ARCHITECTURE

<一句话概述项目解决的问题>

## 概述

<2-3 句鸟瞰视角：项目解决什么问题，核心价值>

## 代码地图

<粗粒度模块及其关系，回答两个问题：
- "X 在哪？"（定位功能）
- "这段代码做什么？"（识别职责）>

### 模块划分

| 模块 | 路径 | 职责 |
|------|------|------|
| <模块名> | <目录路径> | <简短职责描述> |

### 模块关系

<简短描述模块间依赖和调用关系，可选配 ASCII 图>

## 架构不变量

<显式声明重要约束，尤其是"缺少什么">

示例：
- Model 层不依赖 View 层
- 所有 API 调用必须经过中间件层
- 数据库操作只在 Repository 模块

## 层级边界

<描述层/系统间的边界，边界蕴含实现信息>

示例：
- API 层与业务层边界：API 只处理请求解析和响应格式化
- 业务层与数据层边界：业务层不直接访问数据库

## 横切关注点

<跨越多个模块的通用机制>

示例：
- 日志：统一通过 Logger 模块
- 错误处理：使用统一的 Error 类型
- 配置：集中在 Config 模块

## 关键文件

<列出最重要的入口文件、核心模块>

示例：
- `src/main.rs` — 程序入口
- `src/core/engine.rs` — 核心处理逻辑
- `src/api/handler.rs` — HTTP 请求处理

## 扩展阅读

<可选：指向更详细的文档，用相对路径而非链接>

示例：
- 详细设计见 `docs/design.md`
- API 文档见 `docs/api.md`
```

## 生成流程

### AI Agent 执行步骤

```
1. 探索项目结构
   - 运行 tree 或 glob 获取目录结构
   - 识别主要模块和目录

2. 识别核心模块
   - 查看入口文件（main、index、app）
   - 分析关键目录职责
   - 识别模块间依赖关系

3. 发现架构不变量
   - 检查导入关系，发现禁止依赖
   - 分析中间件、代理等隔离层
   - 识别命名模式（如 *Repository、*Service）

4. 生成文档
   - 按骨架模板填写
   - 保持简洁（建议 50-150 行）
   - 不链接，用符号名

5. 验证文档
   - 是否回答"X 在哪？"
   - 是否回答"这段代码做什么？"
   - 是否描述了不变量
```

## 内容取舍指南

### 应包含

| 内容 | 说明 |
|------|------|
| **问题概述** | 项目解决什么问题，用户视角的价值 |
| **模块划分** | 粗粒度模块及职责（不是每个函数） |
| **模块关系** | 依赖方向、调用链 |
| **架构不变量** | 禁止依赖、必须经过的层等约束 |
| **层级边界** | 各层职责边界，边界蕴含实现约束 |
| **横切关注点** | 日志、错误处理、配置等通用机制 |
| **关键文件** | 入口和核心文件名称 |

### 不应包含

| 内容 | 反面案例 |
|------|----------|
| **详细实现** | 每个函数的作用、算法细节 |
| **API 列表** | 每个接口的参数和返回值 |
| **配置项** | 所有配置选项说明 |
| **安装指南** | 安装和部署步骤（放 README） |
| **贡献指南** | 如何贡献代码（放 CONTRIBUTING） |
| **频繁变化的内容** | 当前的具体实现细节 |

## 框架适配

### Codex

| 概念 | 实现 |
|------|------|
| 探索工具 | `list_files`, `read_file`, `search` |
| 生成方式 | 直接写入 `ARCHITECTURE.md` |
| 触发词 | "为项目生成 ARCHITECTURE.md" 或 "写架构文档" |

### Claude Code

| 概念 | 实现 |
|------|------|
| 探索工具 | `Glob`, `Read`, `Grep` |
| 生成方式 | `Write` 工具写入文件 |
| 触发方式 | 用户请求或 skill 加载 |

### 工具映射表

| Codex 工具 | Claude Code 工具 | 用途 |
|------------|------------------|------|
| `list_files` | `Glob` | 获取目录结构 |
| `read_file` | `Read` | 读取关键文件 |
| `search` | `Grep` | 搜索导入关系、依赖 |
| `write_file` | `Write` | 创建 ARCHITECTURE.md |

## 质量检查清单

生成后自检：

| 检查项 | 是否满足 |
|--------|----------|
| 简短（50-150 行） | ✅/❌ |
| 回答"X 在哪？" | ✅/❌ |
| 回答"这段代码做什么？" | ✅/❌ |
| 描述了架构不变量 | ✅/❌ |
| 描述了层级边界 | ✅/❌ |
| 无失效链接 | ✅/❌ |
| 用符号名而非路径链接 | ✅/❌ |
| 不包含频繁变化的内容 | ✅/❌ |

## 一句话总结

> ARCHITECTURE.md = 项目的代码地图，不是详细说明书。
> 核心：简短、稳定、定位功能、识别职责、声明约束。
