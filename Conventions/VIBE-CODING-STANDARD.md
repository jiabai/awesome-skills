# Vibe Coding Standard

适用于 Codex 和 Claude Code 的 AI 代理驱动开发全局规范。

---

## 文件定位

这是一个**元规范**文件，定义"如何让 AI 代理高效驱动软件开发"。

与现有规范的关系：

| 文件 | 定位 | 覆盖范围 |
|------|------|----------|
| `PLANS-UNIVERSAL.md` | 任务执行规范 | 如何写单个 ExecPlan |
| `ARCHITECTURE-TEMPLATE.md` | 架构文档规范 | 如何写 ARCHITECTURE.md |
| `VIBE-CODING-STANDARD.md` | 全局操作系统 | 如何让代理持续高效工作 |

---

## 核心原则（不可协商）

```
Humans steer. Agents execute.
人类引导方向，代理执行代码。
```

**核心约束**：人类注意力是最稀缺资源。所有投入都应增加代理杠杆率。

---

## 三大支柱

| 支柱 | 核心问题 | 目标 |
|------|----------|------|
| **知识管理** | 代理只能看到 repo 内内容 | 让代理能看到一切 |
| **架构约束** | 代理复制现有模式（包括不好的） | 让代理不能乱跑 |
| **开发流程** | 代理吞吐量远超人类注意力 | 让代理高效循环 |

## 一、知识管理规范

### 核心原则

**代理的认知边界**：从代理视角，任何它无法在运行时访问的内容，实际上不存在。

```
┌─────────────────────────────────────────────────────────────────┐
│                    代理可见的知识边界                              │
│  ┌─────────────────┐                                            │
│  │  Repository     │  ← 代理唯一可访问的知识源                    │
│  │  - 代码         │                                            │
│  │  - 文档         │                                            │
│  │  - Schema       │                                            │
│  │  - ExecPlan     │                                            │
│  └─────────────────┘                                            │
│                                                                 │
│  不可见：Google Docs、Slack、会议记录、人类记忆                    │
│  必须编码进 repo 才能被代理使用                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 文件结构规范

```
AGENTS.md (~150行) → 目录/地图，不写百科全书
docs/
├── design-docs/
│   ├── index.md              # 设计文档索引
│   ├── core-beliefs.md       # 核心信念和原则
│   └── *.md                  # 其他设计文档
├── exec-plans/
│   ├── active/               # 正在执行的计划
│   ├── completed/            # 已完成的计划
│   └── tech-debt-tracker.md  # 技术债追踪
├── product-specs/
│   ├── index.md              # 产品规格索引
│   └── *.md                  # 各功能规格
├── references/
│   ├── *.txt                 # 技术参考（LLM友好格式）
│   └── *.md                  # API文档等
├── generated/
│   └── db-schema.md          # 自动生成的文档
├── ARCHITECTURE.md           # 架构地图
├── QUALITY_SCORE.md          # 质量评分追踪
├── SECURITY.md               # 安全规范
└── DESIGN.md                 # 设计规范
```

### AGENTS.md 规范（不可协商）

| 原则 | 说明 | 反面案例 |
|------|------|----------|
| **是地图不是百科** | 指向更深文档，不挤占 context | 写 1000+ 行详细规则 |
| **渐进披露** | 小入口 → 教会代理去哪里找更多信息 | 一次性灌入所有信息 |
| **机械验证** | CI 检查文档新鲜度、交叉链接 | 依赖人工维护 |
| **保持稳定** | 只写相对稳定的内容，频繁变化的放别处 | 每次提交都更新 AGENTS.md |

**AGENTS.md 模板骨架**

```markdown
# 项目名 AGENTS.md

## 快速入口

- 架构：见 `docs/ARCHITECTURE.md`
- 设计规范：见 `docs/DESIGN.md`
- 产品规格：见 `docs/product-specs/index.md`
- 执行计划：见 `docs/exec-plans/active/`
- 技术债：见 `docs/exec-plans/tech-debt-tracker.md`

## 核心信念

<3-5 条不可协商的原则，指向 docs/design-docs/core-beliefs.md>

## 开发流程

<简短的开发流程描述，指向详细流程文档>

## 常用命令

<5-10 个最常用命令>
```

### 必须编码进 repo 的知识类型

| 知识类型 | 原始位置 | 编码位置 | 编码方式 |
|----------|----------|----------|----------|
| 架构决策讨论 | Slack/会议 | `docs/design-docs/` | Decision Log 格式 |
| 产品原则 | 团队共识 | `docs/design-docs/core-beliefs.md` | 明确陈述 |
| API 使用方式 | 外部文档 | `docs/references/` | 复制或摘要（LLM友好格式） |
| 技术债清单 | 人类记忆 | `docs/exec-plans/tech-debt-tracker.md` | 结构化列表 |
| 质量评分 | 会议讨论 | `docs/QUALITY_SCORE.md` | 表格追踪 |

### 知识新鲜度维护

| 机制 | 触发频率 | 执行方式 |
|------|----------|----------|
| **文档园艺代理** | 定期（如每周） | 扫描过时文档 → 打开修复 PR |
| **交叉链接检查** | CI | 验证文档引用是否存在 |
| **结构验证** | CI | 验证 docs/ 目录结构正确 |

## 二、架构约束规范

### 核心原则

**熵的本质**：代理复制现有模式，包括不好的模式。无约束时，架构必然退化。

**约束哲学**：中心强制边界，局部允许自由。边界、正确性、可复现性强制；具体实现自由。

### 分层架构规范

```
每个业务领域固定层次：

┌─────────────────────────────────────────────┐
│                    UI                        │  ← 用户界面
├─────────────────────────────────────────────┤
│                  Runtime                     │  ← 运行时逻辑
├─────────────────────────────────────────────┤
│                  Service                     │  ← 业务服务
├─────────────────────────────────────────────┤
│                   Repo                       │  ← 数据访问
├─────────────────────────────────────────────┤
│                  Config                      │  ← 配置
├─────────────────────────────────────────────┤
│                   Types                      │  ← 类型定义
└─────────────────────────────────────────────┘

依赖方向：只能"向前"（向下）依赖
         Types → Config → Repo → Service → Runtime → UI
         
跨切关注点（auth、telemetry、feature flags）：
         通过单一接口 Providers 进入
         
其他依赖方向 → 机械禁止
```

### 强制约束类型

| 约束类型 | 实现方式 | 示例规则 |
|----------|----------|----------|
| **依赖方向** | 自定义 linter | `UI` 层禁止直接依赖 `Repo` 层 |
| **边界验证** | 结构测试 | 跨切模块必须通过 `Providers` 接口 |
| **文件大小** | linter 规则 | 单文件不超过 500 行 |
| **命名规范** | linter 规则 | Schema 类型必须有特定后缀 |
| **结构化日志** | linter 规则 | 禁止 `console.log`，使用 Logger |
| **数据边界** | linter 规则 | 边界处必须验证数据形状 |

### 约束写入方式

**关键**：linter 错误信息本身就是代理可读的指导。

```yaml
# 示例 linter 配置
rules:
  - id: no-ui-to-repo-dependency
    message: |
      UI 层禁止直接依赖 Repo 层。
      
      正确做法：
      1. UI → Service → Repo（通过 Service 间接访问）
      2. 或使用 Providers 接口
      
      参考：docs/ARCHITECTURE.md#层级边界
```

### 架构不变量示例

| 不变量 | 说明 | 验证方式 |
|--------|------|----------|
| Model 不依赖 View | 数据层不了解 UI | 导入检查 |
| API 经过中间件 | 所有请求经过统一入口 | 路由配置检查 |
| 数据库操作限于 Repo | 业务层不直接 SQL | 导入检查 |
| 边界验证数据形状 | 外部数据必须验证 | 类型检查 |

### 黄金原则

| 原则 | 理由 | 反面案例 |
|------|------|----------|
| **共享工具包优于手写 helper** | 不变量集中，避免重复 | 每个文件写自己的 `formatDate` |
| **边界验证优于 YOLO 猜测** | 代理不能猜测数据形状 | 直接使用未验证的 API 返回 |
| **"无聊"技术优于新奇技术** | 训练集覆盖、API 稳定 | 使用刚发布的前端框架 |
| **自实现优于 opaque 库** | 代理能理解、修改、测试 | 引入 500KB 的工具库 |

## 三、开发流程规范

### 开发循环

```
┌──────────────────────────────────────────────────────────────────┐
│                        开发循环                                    │
│                                                                  │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐       │
│   │ 描述任务 │ → │ 运行代理 │ → │ 创建 PR  │ → │ 代理审查 │       │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘       │
│        ↑                                            │            │
│        │                                            ↓            │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐       │
│   │   合并  │ ← │ 循环满意 │ ← │ 处理反馈 │ ← │ 其他代理 │       │
│   └─────────┘    └─────────┘    └─────────┘    │   审查  │       │
│                                                  └─────────┘       │
│                                                                  │
│   人类介入点：仅在需要判断时                                        │
└──────────────────────────────────────────────────────────────────┘
```

### PR 处理原则

| 原则 | 说明 | 反面案例 |
|------|------|----------|
| **短生命周期** | PR 快进快出，不长期挂起 | PR 存在数周等待审查 |
| **最小阻塞** | Test flake 用后续运行处理 | 单个 flake 阻塞整个 PR |
| **代理对代理审查** | 大部分审查由代理完成 | 依赖人类逐行审查 |
| **修正便宜，等待昂贵** | 快速合并 → 快速修复 | 追求每次 PR 完美无瑕 |

### 代理审查流程

```
1. 代理自审查（本地）
   - 运行测试
   - 检查 linter
   - 验证文档更新
   
2. 其他代理审查（云端）
   - 代码风格
   - 架构边界
   - 测试覆盖
   
3. 人类审查（可选）
   - 产品方向
   - 关键决策
   
4. 处理反馈 → 循环直到所有审查者满意
```

### 熵管理（垃圾收集）

**核心洞察**：技术债像高利贷，小额持续偿还优于一次性大清理。

| 机制 | 触发频率 | 执行方式 |
|------|----------|----------|
| **黄金原则扫描** | 定期（如每日） | 后台代理扫描偏差 → 打开重构 PR |
| **质量评分更新** | 每次里程碑 | `QUALITY_SCORE.md` 追踪各模块质量 |
| **技术债偿还** | 持续小额 | 不积累，每天处理而非数周爆发 |
| **模式清理** | 发现偏差时 | 打开针对性重构 PR → automerge |

**熵管理代理任务示例**

```
每日熵管理任务：
1. 扫描 docs/ 检查过时内容 → 打开文档修复 PR
2. 扫描代码检查违反黄金原则 → 打开重构 PR
3. 更新 QUALITY_SCORE.md 评分
4. 检查 tech-debt-tracker.md 偿还进度
```

## 四、可观测性规范

### 核心原则

**代理自治的前提**：代理需要能自己验证工作，不依赖人类 QA。

### 代理可访问的验证能力

| 能力 | 实现方式 | 用途 | 验证示例 |
|------|----------|------|----------|
| **UI 验证** | Chrome DevTools Protocol | 代理驱动应用、验证 UI 行为 | 点击按钮 → 截图对比 |
| **DOM 快照** | CDP + skills | 获取页面状态、验证渲染 | 检查元素是否存在 |
| **应用隔离** | git worktree | 每个任务独立实例 | 并行任务不互相干扰 |
| **日志查询** | 本地观测栈 + LogQL | 查询启动时间、错误追踪 | `startup_time < 800ms` |
| **指标查询** | 本地观测栈 + PromQL | 验证性能目标 | `latency_p99 < 2s` |
| **Trace 查询** | TraceQL | 分析调用链瓶颈 | 检查 span 时长 |

### 验证 prompt 示例

| Prompt 类型 | 示例 | 代理执行路径 |
|-------------|------|--------------|
| **性能验证** | "确保服务启动完成在 800ms 内" | 查询日志/指标 → 发现问题 → 修复 → 重启 → 再验证 |
| **路径验证** | "这四个关键用户路径的 span 不超过 2 秒" | 驱动 UI → 查询 trace → 分析瓶颈 → 修复 → 再验证 |
| **行为验证** | "点击登录按钮后应显示用户名" | 截图/DOM → 对比 → 修复 → 再验证 |

### 自治能力里程碑

代理端到端驱动一个 feature：

```
┌─────────────────────────────────────────────────────────────────┐
│                      代理自治能力里程碑                           │
│                                                                 │
│  1. 验证当前状态                                                 │
│  2. 复现 bug                                                    │
│  3. 录制失败视频                                                 │
│  4. 实现修复                                                    │
│  5. 验证修复                                                    │
│  6. 录制成功视频                                                 │
│  7. 创建 PR                                                     │
│  8. 处理审查反馈                                                 │
│  9. 合并                                                        │
│  10. 仅在需要判断时请求人类                                       │
│                                                                 │
│  ↑ 这需要大量前期投入（结构、工具、skills）                        │
└─────────────────────────────────────────────────────────────────┘
```
